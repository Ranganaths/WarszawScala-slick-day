Before we dive into slick, let's talk what is wrong with ORM's?

Object-relational impedance mismatch
(The Vietnam of Computer Science)

  • Leaky abstraction - you can feel it those are not objects
  • Very easy to start, very hard to bend it to all needs
  • Complex, complicated, sometimes performance killer
  • Often used incorrectly:
    • Getting the whole object-graph from database
    • Referring to objects not fully fetched, ouside of the persistence context

What's so special about slick?

  • A Functional Relational Mapper for Scala
  • Does not try to mimic DB relations as objects
  • Uses tuples
  • You already know how to write code in it (collections-like API, for-comprehension FTW!)
  • Flexible query composition
  • It's very specific when it hits the database so it's easier to track side-effects
  • It's type safe
  • Embraces SQL and maps it to Scala features

How to operate on them in slick?

Sorry, you can't do this


case class Address( … )
case class Person( …, address: Address )

This is why we have ORM problems

Embrace SQL


SELECT * 
FROM Person 
JOIN Address ON Person.addressId=Address.id;
						

val tupledJoin: Query[(People,Addresses),(Person,Address), Seq]
  = people join addresses on (_.addressId === _.id)

case class PersonWithAddress(person: Person, address: Address)

val caseClassJoinResults = db.run(tupledJoin.result)
  .map(_.map(PersonWithAddress.tupled))
